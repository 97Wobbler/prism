# ADDIE Model

## Overview
ADDIE is a five-phase iterative framework for instructional design: Analyze the learner need and performance gap, Design learning objectives and instructional strategies, Develop the actual content and materials, Implement delivery and track adoption, Evaluate outcomes against objectives and refine. Practitioners use it to move instruction from vague intention ("teach people X") to measurable outcomes and validated impact. The discipline is the relentless demand that each phase produce artifacts that the next phase can build on—and that nothing advances until prior phases have real evidence, not assumptions.

## Analytical Procedure

### Phase 1 — Analyze

1. **Define the performance gap in measurable terms.** Not "employees need to be better at conflict resolution" but "employees in department X take an average of 4 escalations per month to HR; target is <1 per month; root cause is lack of structured negotiation tactics." Document the current state (measurement), desired state (measurement), and the evidence for the gap.

2. **Identify the learner population.** Write:
   - Current role(s) and experience level
   - Prior knowledge relevant to the instruction
   - Constraints (time available, device access, literacy level, language proficiency)
   - Motivation (Why are they taking this? Is it required, incentivized, or intrinsic?)

3. **List the conditions under which learning must occur.** When will they learn? For how long? Will they be in a classroom, online, on the job? Will they have a trainer? Can they loop back and re-access materials?

4. **Determine success metrics.** Define what "closing the gap" looks like in behavioral terms:
   - Performance metrics (what will they do differently?)
   - Transfer metrics (where will they apply it?)
   - Retention metrics (how long must they remember it?)
   - Business metrics (what does success cost in ROI terms, if applicable?)

### Phase 2 — Design

5. **Write learning objectives in measurable form.** Each objective must follow: "After completing this instruction, the learner will be able to [verb] [object] in [context]."
   - Verbs must be observable: identify, construct, explain, apply, evaluate, not "understand" or "appreciate."
   - Objects are specific: "the seven steps of conflict de-escalation" not "conflict resolution techniques."
   - Context specifies when/where: "in a role-play scenario" or "under time pressure" or "without trainer support."

6. **Map objectives to the performance gap.** For each objective, trace back: which gap does this close? If an objective doesn't close any gap, delete it.

7. **Select instructional strategies.** For each objective, specify the method(s):
   - Lecture (knowledge transfer only; no skill-building)
   - Demonstration + guided practice (skill-building with support)
   - Role-play or simulation (realistic context)
   - Problem-based learning (learn by solving)
   - Reflection or peer feedback (consolidation)
   Document why each strategy matches its objective. Generic "variety is good" is not a reason.

8. **Outline content structure.** Sketch the sequence: which topics first, which depend on others, where are the checkpoints? Identify prerequisites.

9. **Draft assessment strategy.** Specify:
   - Formative assessments (during learning, for feedback): What will tell the learner they're on track?
   - Summative assessment (at end, for certification): What proves they've met the objectives?
   - For each: Is it performance-based (they do the task) or knowledge-based (they answer questions)? Why that choice?

### Phase 3 — Develop

10. **Produce the instructional materials.** This includes content, visuals, simulations, role-play scripts, quizzes, job aids, etc. Each must map explicitly to one or more learning objectives.

11. **Build assessments.** Create the tests, role-play rubrics, or performance checklists defined in Phase 2. Pilot them with 3-5 learners from the target population. Refine items that confuse people or where the answer key doesn't match actual learner performance.

12. **Conduct internal review.** A subject-matter expert and an instructional designer unfamiliar with the course should review:
    - Does each activity require only the prerequisite knowledge listed in Phase 2?
    - Are instructions clear enough that a learner could attempt the activity without live help?
    - Do assessments actually measure the stated objectives, or do they measure something else?

### Phase 4 — Implement

13. **Pilot with a subset of the learner population.** Run the course with 5-20 learners from the actual target group (not friendly insiders). Observe:
    - Time to completion (does it match the estimate?)
    - Dropout or bailout points (where do learners give up?)
    - Confusion (which instructions or activities cause backtracking or wrong answers?)
    - Engagement (which parts do learners want to repeat, and which do they skip?)

14. **Document implementation conditions.** If the course is delivered live, what does the trainer need? If online, what is the learning management system (LMS)? Are there devices or bandwidth constraints that affect access?

15. **Deliver at scale and track adoption.** Launch to the full population. Record:
    - Completion rates (what % started? Completed?)
    - Time to completion (by cohort, by role, by prior experience)
    - Dropout points (where does the attrition happen?)
    - System performance (did the LMS crash? Did videos buffer?)

### Phase 5 — Evaluate

16. **Measure immediate learning outcomes.** Did learners pass the assessments? What was the distribution of scores? Disaggregate by learner characteristics (role, experience, cohort) — if one subgroup consistently scores lower, that's a signal of misaligned design.

17. **Measure behavior change (transfer).** 2-4 weeks post-training, observe or collect evidence on whether learners are actually applying what they learned:
    - For the conflict resolution example: are escalations down? By how much?
    - Did the gain persist or fade?
    - Are there contexts where transfer failed (e.g., they apply it in one-on-one but not in group settings)?

18. **Compare outcomes to the success metrics defined in Phase 1.** Did you close the gap? By how much? Is the ROI positive (if you calculated one)?

19. **Identify root causes of any shortfall.** If outcomes missed the target:
    - Was the gap analysis wrong (the problem wasn't what you thought)?
    - Was the design misaligned (the course was good, but it didn't address the real barrier)?
    - Was the delivery broken (the course was fine, but people didn't have time to apply it)?
    - Was the assessment wrong (you measured the wrong thing)?

20. **Plan iteration.** Document what to change in the next cycle. Prioritize by impact (does fixing this close the gap?) and feasibility (can we fix it in the next 3 months?).

## Evaluation Criteria

| Check | Pass |
|---|---|
| Performance gap is stated as a measurable before/after | Y/N |
| Learner population includes experience, constraints, and motivation | Y/N |
| Learning objectives are stated with observable verbs and specific context | Y/N |
| Each objective maps to at least one gap and one design strategy | Y/N |
| Instructional strategies are justified, not generic | Y/N |
| Assessments distinguish knowledge from performance | Y/N |
| Pilot involved 5+ learners from actual target population | Y/N |
| Implementation tracked completion, dropout, and time-to-completion | Y/N |
| Evaluation measured both immediate learning and behavior change | Y/N |
| Root cause analysis was applied to any gap in outcomes | Y/N |

## Red Flags

- Learning objectives include words like "understand," "appreciate," "learn," or "be familiar with." These are not observable; start over.
- The Analyze phase skipped learner motivation or constraints. Without knowing *why* people are here and *when* they can attend, the design will misfire.
- Instructional strategies are the same for every objective (all lecture, all role-play). The real work is matching strategy to outcome.
- No pilot with actual learners from the target population. Using friendly colleagues or subject experts will hide usability problems.
- Evaluation stops after a pass/fail test score. Transfer is where the real verdict happens — if learners don't apply it, the instruction failed, regardless of test scores.
- Root cause analysis is skipped or generic ("we'll make it more engaging"). The point is to isolate which *phase* went wrong so you know where to iterate.
